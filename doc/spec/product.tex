\section{IML-Teilgraphen und Selektionen}
Hinsichtlich der logischen Zusammenfassung von Knoten und Kanten unterscheidet
GIANT IML-Teilgraphen Selektionen.

  \subsection{IML-Teilgraphen}
  Ein IML-Teilgraph beschreibt eine Knoten und Kantenmenge aus dem IML-Graphen,
  bei der jede Kante hat einen Start- und einen Zielknoten hat.\\
  Die Knoten und Kanten von IML-Teilgraphen können in Anzeigefenstern
  hervorgehoben werden, des weiteren können die Knoten und Kanten von
  IML-Teilgraphen in Anzeigefenstern layouted und dargestellt werden.\\
  Die IML-Teilgraphen sind aber völlig unabhängig von den Anzeigefenstern und
  enthalten insbesondere keine Layoutinformationen für ihre Knoten und Kanten.

  \subsection{Selektionen}
  Selektionen stellen die lokale Variante von Knoten- und Kantenmengen dar.
  Sie sind immer einem festen Anzeigefenster zugeordnet.\\
  Selektionen können bleibige Knoten und Kanten umfassen, ohne dass diese einen
  Teilgraphen bilden müssen.\\
  Jeder Knoten und jede Kante einer Selektion muss auch im Anzeigfenster
  visualisiert sein.

\section{Anzeigefenster}
Anzeigefenster sind die Fenster von GIANT, in denen eine benutzerdefinierte
Auswahl von Knoten und Kanten visualisiert wird. Es kann beliebig viele
Anzeigefenster geben und jedes Anzeigefenster kann beliebig viele Selektionen
haben.

  \subsection{Pins}
  Da selten alle zu einem Anzeigeinhalt gehörenden Knoten und Kanten insgesamt
  sichtbar dargestellt werden können (da es zu viele sind), kann sich der
  Benutzer zu jedem Anzeigefenster eine Liste von Pins anlegen. In den Pins wird
  jeweils die Position des sichtbaren Anzeigeinhaltes und die Zoomstufe
  gespeichert, so dass zu beliebigen Zeitpunkten der jeweilige sichtbare
  Anzeigeinahlt rekonstruiert werden kann.

\section{Anfragen}
Eine vielseitige Anfragesprache stellt nahezu die gesamte Funktionalität
von GIANT zur Verfügung und kann insbesondere auch zum Aufruf via Kommandozeile
genutzt werden. Die drei Schritte des anschließend beschriebenen 
\gq{Drei-Stufen-Konzeptes} können können über diese Anfragesprache auch \gq{auf 
einen Schlag} erledigt werden.


% ==============================================================================
\section{Drei-Stufen-Konzept}

Die im Folgenden beschriebenen drei Schritte sollen dem Benutzer die Möglichkeit
bieten, von den Knoten und Kanten des IML-Graphen ausgehend geeignete
Teilgraphen in Anzeigefenstern zu visualisieren. Von der
Gestaltung der UseCases her, werden diese drei Schritte sequentiell nacheinander
ausgeführt, dies ist aber nicht zwingend nötig.


\begin {enumerate}
  \item Erzeugen geeigneter IML-Teilgraphen, d.h Auswahl geeignter Knoten und
  Kanten aus dem IML-Graphen mittels der Anfragesprache.
  
  \item Ableiten weiterer IML-Teilgraphen aus bereits bestehenden
  IML-Teilgraphen (mittels der Anfragesprache, z.B. Mengenoperationen).

  \item Einfügen dieser IML-Teilgraphen in ein Anzeigefenster unter Anwendung
  eines Layoutalgorithmus. Innerhalb des Anzeigefensters ist der eingefügte 
  IML-Teilgraph dann als Selektion weiterhin greifbar.
\end {enumerate}

\section{Fehlerverhalten}

GIANT soll sich bezüglich der Fehlerbehandlung konsistent verhalten.
Das im folgenden beschriebene Fehlervarhalten gilt daher pauschal für
das gesamte System, Abweichungen von diesem Verhalten werden
gegebenenfalls an der entsprechenden Stelle (z.B.\ bei dem jeweiligen
UseCase) explizit spezifiziert.\\
Fehlermeldungen werden grundsätzlich über den STANDARD-FEHLERDIOLOG
ausgegeben. Ist dies nicht möglich, wird eine entsprechende Fehlermeldung
über Kommandozeile ausgegeben.

  \subsection {Ungültige oder unvollständige Eingaben bei Dialogen}
  Eingaben in Dialoge etc. werden immer dann geprüft, wenn der Benutzer
  die Eingaben in dem gesamten Dialogfenster z.B. durch klicken auf einen
  \gq{OK-Button} bestätigt.\\ 
  Stellt das System dann eine unzulässige oder fehlende Eingabe fest, so
  erscheint der STANDARD-FEHLERDIALOG mit einer entsprechenden Mitteilung
  zur Fehlerursache.\\
  Nach Schließen des STANDARD-FEHLERDIALOGES kehrt das System zu Stelle
  im Ablauf zurück, die dem Zustand vor der Fehlerprüfung entspricht, also
  typischerweise zu dem Dialog in dem die fehlerhafte Eingabe getätigt 
  wurde.\\
  Die zuvor in den entsprechenden Dialogen getätigten Eingaben des Benutzers
  (die Eingaben die den Fehler verursacht haben) bleiben erhalten.

  \subsection {Ungültige oder unvollständige Eingaben bei Kommandozeilenaufruf}
  Der Start von GIANT wird abgeborchen. Auf der Kommandozeile wird eine
  entsprechende Meldung mit Hinweisen zur Fehlerursache ausgegeben.

  \subsection {Fehler beim Einlesen von Dateien}
  Aufgrund der Vielzahl der hierbei möglichen Fehlern (technische Defekte etc.)
  kann das Verhalten grundsätlich nicht spezifiziert werden.
  Generell ist folgendes Verhalten angestrebt:
  
  \begin{enumerate}
    \item Können zum Betrieb von GIANT nötige Dateien nicht an der 
          dafür vorgesehenen Position gefunden werden, so wird das 
          System mit einer entsprechenden Fehlermeldung beendet.

  \end{enumerate} 

  Ein anderes Verhalten (z.B. die Eingabemöglichkeit für einen alternativen
  Pfad) wird gegebenenfalls an entsprechender Stelle explizit spezifiziert.
